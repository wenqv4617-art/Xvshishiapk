### 2. 新增核心函数与逻辑解说

#### 2.1 财务隔离
*   `getBalanceKey()` / `getLedgerKey()`：根据当前 `localStorage` 中的 `active_me_id` 为每个用户人设动态计算出隔离的专属 LocalStorage 键名，从而实现在切换面具时余额与账单流水的独立变更。

#### 2.2 朋友圈手势交互
*   `startCommentLongPress(el, commentId, event)`：挂载触控和鼠标的长按（800ms）监听，触发评论行背景变红高亮并显示垃圾桶 SVG，且通过 Pointer Events 样式规则阻断了 SVG 内部 Path 元素对点击事件的干扰。
*   `handleCommentClick(momentId, commentId, commenterName, event)`：对长按状态进行检查。如果已被激活，点击行为将安全折叠恢复原状而不再触发回复模态窗，解耦了删除状态与回复状态。

#### 2.3 仿真时间与流逝计算
*   `getMessageDisplayDate(msg, sess)`：在渲染每条消息时调用。若单聊设置中关闭了“时间感知”，系统将根据虚拟起始时间（`customTimeData`）和设定保存时戳（`customTimeSavedAt`）累加该消息发送时的相对物理差额，计算出拟真场景下的显示时间。
*   `formatWeChatTime(date, relativeToDate)`：实现微信拟真的多态时间表示。根据传入的目标时间与相对时间（Simulated Now）的对比，按天、昨天、当年、跨年等不同格式渲染出诸如 `12:00`、`昨天 15:30` 或 `2026年1月1日 12:00` 样式的灰色系统时间戳。

#### 2.4 多物理手势解耦与防抖
*   `safeOpenMomentFromShare(momentId, event)`：为朋友圈转发卡片引入了 500 毫秒延时防抖。若用户在 0.5 秒内双击消息气泡呼出操作菜单，定时器将自动在双击事件中被清空清除，避免了“单击跳转朋友圈”与“双击呼出编辑菜单”之间的事件抢占冲突。

#### 2.5 消息撤回状态机
*   `toggleRecallContent(el, event)`：撤回的消息将转化为完全居中、去 bubble 耦合的灰色系统通知条。点击“查看”按钮可就地向下展开/收起已被撤回的原始消息本体，避免了系统 Alert 弹窗对用户操作的打扰。

#### 2.6 长按 Emoji 贴纸反应
*   `startBubbleLongPress(msgId, bubbleEl, e)`：检测气泡长按达 2 秒触发。内部会动态抓取当前会话最近的 20 条消息（即 10 轮交互），确保该消息在 20 条区间内才予以激活。
*   `saveReaction(msgId, emoji)` / `removeReaction(msgId, event)`：提供表情反应贴纸的覆盖和一键点击剥除功能，并在剥除时进行 `stopPropagation` 阻止底层气泡事件冒泡。

### 3. 技术路线与实现架构分析

```
                        ┌──> [app_prompts.js] (Prompt 拼装与动态条件注入)
                        ├──> [app_chat.js] (时间随动计算、长按/双击手势及状态变更)
[用户交互 / 大模型反应] ────┼──> [app_moments.js] (评论长按机制、解耦同组反应范围)
                        ├──> [app_chat_quote.js] (滑动手势与选择器解耦判定)
                        └──> [app_wallet.js] (人设随动钱包余额多态存储)
```

#### 3.1 微信多态时间推算技术路线
为了在单聊中完全支持“关闭时间感知（使用设定时间）”和“开启时间感知（跟随真实时钟）”两种模式，系统设计了 **相对流逝比例推演机制**：
*   当关闭时间感知时，系统并不使用当前真实的系统时钟作为显示时戳，而是通过计算 `Date.now() - sess.customTimeSavedAt` 得到该对话在现实世界中所流逝的毫秒差额。
*   将该差额直接累加到设定的基准时间（如 2026-01-01 12:00）上，得出一个随真实时空 1:1 正常流逝演进的虚拟场景时间。
*   主列表（Session List）的排序则永远依赖消息的真实物理时戳，从而保障了会话列表检索性能与聊天视觉场景体验的统一。

#### 3.2 自带事件阻断的绝对定位 UI 架构
为了在不使用额外第三方库的情况下，实现气泡长按 Emoji 选择器、气泡原地撤回展开、已贴表情点击移除等复杂手势，系统贯彻了**作用域物理隔离设计**：
*   撤销了在 `.msg-bubble` 外层套用 `.msg-content-wrapper`（具有 display: flex）的设计。转而让具体气泡卡片本身承担 `position: relative`，从而保留了全局 `style.css` 和 `chat.css` 的所有原生 padding、margin 和 avatar 间距属性，消除了气泡对折缩窄和间距紊乱的异常。
*   利用 `e.target.closest` 技术对 Quotation 向左滑动进行始发点探测。如果触控手势是从 `.bubble-emoji-picker` 或 `.bubble-attached-emoji` 内部发起的，滑动引用模块会在初始化阶段予以直接拦截并 `return`，从而彻底解决了表情选择器横向滑动选择与气泡左滑引用之间的冲突。

### 1. 更新功能板块与技术细节

#### 1.1 全局 PWA 自研对话框与消息组件系统 (Toast & Modal Dialog Engine)
*   **开发背景**：原生浏览器的 `alert()`、`confirm()` 以及 `prompt()` 灰色弹窗会严重破坏虚拟手机的扮演沉浸感 [1]。
*   **重构方案**：
    *   在 `app_chat.js` 头部主动注入了一套具有毛玻璃质感、微白色发光描边、深色半透明悬浮条的 CSS 与 JavaScript 双端联合组件。
    *   **接口封装**：
        *   `showToast(msg, duration)`: 在底部 120px 处呈现 0.2s 渐入淡出的半透明悬浮条 [1]。
        *   `showCustomAlert(title, message, callback)`: 全屏卡片式错误及结果反馈 Dialog [1]。
        *   `showCustomPrompt(title, defaultValue, callback)`: 卡片式数值输入 Dialog（替换充值、提现时粗暴的 `prompt` 录入）[1]。
        *   `showCustomConfirm(title, message, onConfirm)`: 带有“取消/确认”双通道的物理删除、归档确认卡片 [1]。
    *   **覆盖板块**：已无损重构并覆盖至**单聊专属设置保存**、**API 请求异常拦截**、**朋友圈动态转发/发送空内容提示**、**微信零钱充值与提现账单**、**朋友圈 AI 自发动态生成**、**档案库及 NPC 新建删除**、以及**深谈卡片回溯 Reroll**。

#### 1.2 时序不退焦发送与回车上屏系统 (Keyboard Focus & Enter Send)
*   **开发背景**：之前点击发送消息按钮时，输入框由于失去焦距（Blur）会导致移动端软键盘收起、弹回，影响多句连发的爽快感 [1]。
*   **重构方案**：
    *   **防抖不退焦**：在发送按钮上利用 `onmousedown` 的 `e.preventDefault()` 阻止输入框失去焦点 [1]；并在 `onclick` 上屏时，在逻辑尾部进行 `dialogInput.focus()` 的显式回焦兜底 [1]。
    *   **回车发送**：在“系统设置”的“桌面美化设置”中追加了 **`回车上屏 (Enter 发送)` 开关** [1]，状态利用 LocalStorage 持久化保存 [1]。在开启后，按 `Enter` 键会自动触发上屏且阻断默认换行，同样保持键盘不收回 [1]。

#### 1.3 大模型中断请求网关 (Request Abortion via AbortController)
*   **开发背景**：网络超时或模型推理过慢（特别是思考型长上下文推理）时，用户往往只能等待而无法干预，容易造成页面假死 [1]。
*   **重构方案**：
    *   引入了标准 `AbortController` 并在 `app_chat.js` 顶部声明了 `onlineAbortController` 与 `offlineAbortController` 中断控制量 [1]。
    *   **动作联动**：线上和线下发送请求后，“获取回复（✨）”按钮会切换成**浅红色的圆角停止方块图标** [1]；此时再次点击该停止按钮，将立即触发 `.abort()` 中断当前的 fetch 神经链接 [1]。
    *   **容错捕获**：在 catch 分支中精准识别 `AbortError` 并予以吞掉 [1]，不再报出长串的 HTTP 报错卡片，而是弹出一行温和的 Toast：「`当前请求已终止`」，同时瞬时将按钮图标和页头 typing 状态还原 [1]。

#### 1.4 拟真物理按压弹性回弹动效 (Two-stage Haptic Long Press)
*   **开发背景**：气泡长按唤出反应贴纸选择器时没有视觉过渡，反馈不够生动灵敏。
*   **重构方案**：
    *   利用两个并行的 `setTimeout` 构造了时间差：按压达到 **1.0秒（第一阶段）** 时，气泡上平滑附加 `.bubble-longpressing` 样式（`scale(0.95)` 微收缩）[1]；按压达到 **1.3秒（第二阶段）** 时，气泡瞬间复原尺寸，并唤出表情贴图选择器 [1]。
    *   中途松手、挪开指针等行为会调用 `cancelBubbleLongPress`，立即清除定时器并弹回原有尺寸，还原了类似于原生 iOS 触觉按压的深度回弹手感 [1]。

#### 1.5 表情反应面板选择器的物理对齐约束 (Reaction Picker Alignment)
*   **重构方案**：为了贴合微信/手 Q 在对话气泡上的视觉偏侧，调整了表情选择器的对齐方案。通过在 `showEmojiPicker` 内读取 bubble 节点的类名：如果为己方发言（`.self`），则选择器采用右对齐（`right: 0`）；若为对方发言（`.other`），则采用左对齐（`left: 0`）[1]。

#### 1.6 语法阻塞自愈修复
*   **问题排查**：在之前的组件合并过程中，头部多媒体场景卡片的 CSS 模板字符串反引号在中途被提早闭合了，导致后续的 `.pwa-toast` 等大段 CSS 裸码泄露至 JavaScript 全局执行域中，触发了 JS 语法解析器阻断 [1]。
*   **修复方案**：已重新将 `multimediaStyle.textContent` 修正为单一无缝的反引号字符串结构，保障了 `app_chat.js` 及其余协同组件的正常加载与评估 [1]。

### 2. 新增核心功能与逻辑解说

#### 2.1 PWA 网页/安卓真机避让机制 (防重叠消减)
*   **痛点**：由于系统 WindowManager 悬浮窗置于最顶层，当在应用内部激活桌宠时，网页 DOM 节点与真机悬浮窗会发生视觉上的重叠重合 [1]。
*   **重构方案**：在 `app_desktop_pet.js` 的生命周期入口进行严格嗅探。当处于 Android 外壳内（`window.AndroidMCP` 存在），**在物理上彻底阻断网页 DOM 的创建**；即使因 WebView 异步生命周期漏网生成，渲染时也会执行 `remove()` 强制彻底销毁，**100% 将渲染和手势操作权转交安卓原生 Windows 系统托管**，消除了双重桌宠并存的异常 [1]。

#### 2.2 动作状态机与多态隔离机制
*   **多状态支持**：桌宠引入 `default`、`happy`、`sad`、`angry`、`hesitant`、`wash`、`eat`、`sleep`、`watch` 共 9 种可自由上传的动作状态。
*   **状态与角色完全解耦**：彻底废弃 LocalStorage，数据全部安全存储于 IndexedDB `desktop_pets` 表中，每个角色状态相互隔离。只有显式开启某角色桌宠，才会自动将其他角色设为禁用并切换 [1]；即使在主程序中**切换到其他角色的聊天页，或者退出 App**，全局活跃的那个角色桌宠也会持续稳定浮现在手机系统桌面上，体验非常连贯 [1]。

#### 2.3 真机全局高精度后台定时发信调度引擎
*   **Kotlin 调度弊端**：原本在 Kotlin 层发起的自动轮询，无法访问 PWA 层复杂的 RAG 总结、会话世界书以及面具核心人设，导致发出的消息质量低下、缺乏上下文感知。
*   **升级方案**：将发信心跳定时器全面收拢于 JavaScript 层。在 `app_desktop_pet.js` 中建立全局每 30 秒轮询的时间片扫描。在前台服务（WakeLock）保活机制下，JS 线程即使在手机屏幕彻底锁屏后依然能够正常运行：
    *   独立扫描各角色的自动发信时间间隔（各角色时间片相互独立、不干扰，如 A 设 10 分钟，B 设 2 分钟） [1]；
    *   达到差额时，**在后台安全调用 WebView 引擎拼装该会话最深处的 RAG 核心记忆与挂载世界书**，生成极高水准的主动发起话题微信内容 [1]；
    *   消息生成后自动入库。App 若处于后台，联动状态栏投射标准的系统通知，桌宠同步冒泡“有人来信。”，构建了极为拟真的微信后台来信链 [1]。

#### 2.4 真机系统桌面不打扰双击交互
*   为了防止拖拽桌宠移动时误触发点击，在 `AndroidMcp.kt` 中设计了手势位移微调过滤（位移差小于 15px 才判定为点击 [1]）。
*   双击悬浮窗时，原生端会将 App 调入后台，延迟 300 毫秒后直接向 WebView 注入执行 `desktopPetSystem.handleDoubleClickBackground()` [1]。大手机的逻辑计算和冒泡渲染直接在后台安静完成，**App 界面绝不弹出打扰用户**，实现了和真实手机桌宠一样的零跳出感交互 [1]。

## 二、 遇到的技术难点、瓶颈与自愈性解决方案

在合并和扩展两个模块时，我们经历了一系列底层的物理层级和逻辑状态冲突。以下是针对这些难题的攻坚记录：

### 2.1 难点一：DOM 层叠上下文引起的设置页“假性消失”与点击穿透
*   **问题现象**：在最初的排版设计中，专注设置面板（`#win-focus-setup`）作为独立应用被直接挂载在 `index.html` 的全局 `body` 底层。当用户在聊天框（`#win-chat`）中点击展开栏的“专注”按钮时，即使为面板追加了 `.active` 激活类名，面板依然被盖在聊天层下方不可见，只有在退出单聊、返回主列表后才能显现；且在移动端按压时极易触发底层会话列表的点击穿透。
*   **成因剖析**：这属于典型的 **CSS 层叠上下文（Stacking Context）混乱**。因为聊天窗口（`#win-chat`）自身是一个带有特定 z-index 的 `.app-window`，且在打开单聊时处于活跃渲染状态。挂载在 `body` 根部、同属于 `z-index: 1000` 的专注面板无法突破 `#win-chat` 已建立的层叠边界，故而被物理遮挡。
*   **解决方案（DOM 物理套袋原则）**：
    我们将 `#win-focus-setup` 与 `#win-focus-active` 面板物理剪切并**嵌套移植到 `#win-chat` 容器最末尾，作为其子元素**。
    这样做，专注面板在被激活时直接继承并重写了 `#win-chat` 内部最顶层的 Stacking 属性，无需改变原有的过渡动画即可直接平滑滑入覆盖在仿真聊天窗口之上，完美消除了层级遮挡与点击穿透。

### 2.2 难点二：大二进制 Blob 在 JSON 序列化中的 `[object Object]` 损坏
*   **问题现象**：专注配置支持上传本地图片作为沉浸式背景。然而，当用户点击保存并重新载入配置后，上传的图片丢失，数据库中的 `avatarUrl` 变成了难以解析的 `"[object Object]"` 字符串，导致背景加载失败。
*   **成因剖析**：专注环境的配置参数（`activeFocusConfig`）在数据库中是作为 JSON 字符串整体存储的。当用户上载本地物理图片时，`onchange` 产生的 `state.tempBlob` 存放的是一个原生的 `File` 对象。对包含 `File` / `Blob` 对象的配置执行 `JSON.stringify()` 时，由于 JSON 规范无法对二进制 Blob 成员执行标准转换，这些成员被解析为 `"{}"`，而在特定环境下赋值时又被转换为隐式的 `"[object Object]"` 字符串，从而导致配置损坏。
*   **解决方案（同载入期 Base64 字符串化）**：
    我们将本地文件选择器的 `onchange` 响应重构为 **FileReader 实时转译模式**：
    ```javascript
    fileInput.onchange = (e) => {
      if (e.target.files.length > 0) {
        const file = e.target.files[0];
        const reader = new FileReader();
        reader.onload = (event) => {
          state.tempBlob = event.target.result; // 这里直接读取并保存为 Base64 Data URL 纯文本字符串
          document.getElementById("focus-config-avatar").value = "[本地上传图片]";
        };
        reader.readAsDataURL(file);
      }
    };
    ```
    转化为 Base64 Data URL 文本字符串后，其天然对 JSON 友好，可以直接进行序列化存储。在渲染层级调用 `resolveAvatar()` 时，该 Base64 字符串又会被浏览器无缝识别为图片源，从底层彻底避开了 Blob 与 JSON 序列化之间的结构冲突。

### 2.3 难点三：极角计算与跨域防断崖式的指示表转盘手势控制
*   **问题现象**：为了提升交互质感，我们尝试将时长的拖动条重构为“拨动圆盘”。但在使用 `Pointer Events` 监听用户在圆环上的极角偏移（Math.atan2）来控制旋转角度时，经常在拖拽到 $180^\circ$ 到 $-180^\circ$ 边界时出现数值“断崖式突跳”，设定分钟从 5 瞬移至 120。
*   **成因剖析**：极角函数 `Math.atan2(y, x)` 的返回值是一个处于 $[-\pi, \pi]$ 区间的弧度值。当用户顺时针旋转跨越极轴左侧中线时，角度会发生突变。若直接用绝对差值乘以系数累加，就会产生数值跳跃。
*   **解决方案（跨阈增量自愈）**：
    在 `pointermove` 中，我们计算两帧之间的极角差值 `angleDiff`，并对其进行快速临界对齐：
    ```javascript
    if (angleDiff > Math.PI) angleDiff -= 2 * Math.PI;
    if (angleDiff < -Math.PI) angleDiff += 2 * Math.PI;
    ```
    通过累加这个修正后的帧间微增量（`angleDiff`），我们得出了一个平滑无波折的分钟变化（`minDelta`），再将总分钟数安全 Clamp 在 `[5, 120]` 限制范围内，并根据分钟比例计算转盘最终对应的 transform 的 rotate 角度，使物理阻尼感极其舒适自然。

### 2.4 难点四：本地 TXT 书籍导入的多编码乱码崩溃
*   **问题现象**：在使用阅读器上载本地 TXT 小说时，部分文件在 UTF-8 编码下加载正常，但换成 GBK/GB2312 编码的文件后，正文会大面积出现以符号 `\uFFFD`（即 ``）为特征的密集乱码，甚至导致章节切割正则匹配失败，导致系统崩溃。
*   **成因剖析**：JavaScript 的 FileReader 的 `readAsText()` 默认使用 `UTF-8` 进行译码。若物理文件底层使用了国内 Windows 环境常见的 GBK 字符集，多字节编码被错误切分即会形成无法识别的非法占位标记。
*   **解决方案（双向编码自愈检测机制）**：
    我们为文件解析流注入了**断言探测自愈模块**。首先以 UTF-8 方式读取前导字节流并分析内容：
    ```javascript
    reader.onload = async (e) => {
      const rawResult = e.target.result;
      // 乱码探测断言：检查是否包含 UTF-8 解码失败特有的占位块
      if (rawResult.includes("\uFFFD") || rawResult.includes("")) {
        const gbkReader = new FileReader();
        gbkReader.onload = async (gbkEvent) => {
          await saveImportedBook(gbkEvent.target.result, fileName, ext);
        };
        gbkReader.readAsText(file, "GBK"); // 强制回退为 GBK 进行二次健康加载
      } else {
        await saveImportedBook(rawResult, fileName, ext);
      }
    };
    ```
    通过捕获这两种最常见的乱码标志，实现了对本地导入文本编码的百分之百无感知自适应解析。

### 2.5 难点五：触控事件穿透与伴读弹层手势抢占
*   **问题现象**：在阅读器的阅读房间（`#win-reading-room`）内，用户需要双击段落触发 AI 伴读生成气泡。但在移动设备上，双击操作经常会同时触发底栏控制菜单的滑入/滑出。此外，伴读生成的圆点小气泡（`.para-comment-bubble-trigger`）在点击展开吐槽卡片时，也会误触发气泡背后的段落长按动作。
*   **解决方案**：
    1.  **菜单调配隔离**：在控制底栏展示开关 `toggleReadingMenuBar(event)` 中，加入了高敏感拦截：
        ```javascript
        if (event.target.closest(".para-comment-balloon") || event.target.closest("button") || event.target.closest(".para-comment-bubble-trigger")) {
          return; // 释放特定功能区域的点击穿透
        }
        ```
    2.  **气泡事件阻断**：为伴读控制圆点气泡的展开绑定了专有的 `onclick` 处理，在入口直接调用 `event.stopPropagation()` 和 `event.preventDefault()`，彻底切断了点击流在事件树上的向上冒泡，消除了多重手势的响应冲突。

## 三、 开发中的工程化思考

在此次迭代的攻坚中，我们总结出以下三条在客户端单页应用中行之有效的架构设计理念：

1.  **“多态财务随动”与“沙箱数据隔离”**
    我们坚持人设随动原则。当切换“我的人设（`active_me_id`）”时，底层的余额及账单、已读书架、统计时长均应自动改变，而无须修改底层的 IndexedDB Schema。这种纯粹根据当前 localStorage 的 active ID 进行前置键名合成的设计（如 `wallet_balance_v1_[activeMeId]`），大大降低了复杂联表查询开销，也保证了面具之间隐私隔离的绝对干净。
2.  **“零写入”视图清洗机制**
    例如“HTML 互动舱”的一键纯净渲染、伴读书评气球的挂载和销毁，都不在核心数据库中存储可能损坏的原生 DOM 结构。我们始终在只读内存（RAM）状态或 DOM 树动态节点中维护这部分临时 UI 树。即使大模型返回的代码发生了中断或损坏，只要刷新页面，系统就能无损自愈，不会弄脏持久化数据库的数据，抗灾容灾能力极强。
3.  **自注册防御挂载锁**
    在 PWA 环境中，静态脚本往往会被 Service Worker 异步提前缓存并执行。若在 DOM 尚未重绘就绪时强行注册 `document.getElementById`，会因为捕获到 null 导致脚本崩溃。因此，在 `app_chat_focus.js` 和 `app_reader.js` 尾部，必须统一引入自愈式就绪注册：
    ```javascript
    if (document.readyState === "loading") {
      document.addEventListener("DOMContentLoaded", initSystem);
    } else {
      initSystem();
    }
    ```
    这套自注册锁已成为该手机系统后续进行各类迷你应用、小部件开发的核心规范。

### 2. 新增核心逻辑与技术路线解说

#### 2.1 极角泛化标签解析与格式自愈 (`parseTextTag` & `cleanTagOutput`)
*   **开发背景**：小型模型在生成文字标签（Text Tag）时，极易带出额外的方括号或将标签头错写为英文、半角、带冒号等异形，甚至直接将标签文字本身混入正文中输出。
*   **重构方案**：
    *   `parseTextTag(text, tagChinese, tagEnglish)`：使用非捕获组与正向断言，可自适应匹配类似 `[标题] xxx`、`【TITLE】: xxx`、`标题：xxx` 等格式，并抓取内容直至下一个标签出现 [4]。
    *   `cleanTagOutput(text)`：在每次解析出片段后自动进行**后置物理清洗（Post-clean Strip）**，利用高精准正则抹除可能溢出到正文最前面的中英文标签头残余，确保渲染到 UI 上的文字绝对干净 [4]。

#### 2.2 查手机纯净背景提炼器 (`buildCheckPhoneBasePrompt`)
*   **开发背景**：原本直接调用 `buildGlobalSystemPrompt`，使大模型带入了 WeChat 线上的聊天规则、微信引用、MSG_ID 规则，导致在生成日记和备忘录时频频“出戏”出现线上打字痕迹。
*   **重构方案**：自建专用隔离 Prompt，只提取 Character 设定、User 设定与他们之间的亲疏纽带。写入**绝对隔离命令**：明确告知模型这并非在线聊天，而是生成本地日记，**强制锁定 Char 为第一人称（“我”）**，User 表现为（“TA”或专属微信备注昵称），保证人设绝对不反转 [1]。

#### 2.3 家电操控 API 动态质问反应 (`triggerCharReaction`)
*   **重构方案**：摒弃了原本从静态 JSON 数组中抽取的死板对白。改为在用户操控任何空调、电灯等家电后，合成设备被控场景的专属 Prompt 调用接口，命令大模型在 25 字内根据当前角色人设产生一句给 User 的纯文字发信对白（杜绝任何括号动作和引用标签）并瞬间同步上屏 [2, 3]，主聊天联动极其自然。

### 4. 踩过的坑及解决对策 (PAIN POINTS & SOLUTIONS)

#### 3.1 坑一：异步 IIFE 中 `let` 变量重定义引发的 `SyntaxError`
*   **踩坑现象**：上一版代码在 IIFE（立即调用函数表达式）最顶部为了防止 TDZ（临时死区），使用了 `let openPhone;` 进行变量声明；但在文件后续又使用了标准的 `async function openPhone() {}` 进行重复定义。这导致 V8 引擎直接抛出 `Uncaught SyntaxError: Identifier 'openPhone' has already been declared`，造成整份 JS 脚本直接无法解析，查手机按钮完全点不动。
*   **对策**：彻底移除了顶部所有的冗余 `let` 声明。由于在 JavaScript 编译期，**标准的 `function` 声明天然拥有声明提升（Hoisting）特权**，引擎会自动将它们注册在闭包作用域最顶端，在完美规避了 TDZ 异常的同时，也让函数彼此间的异步相互调用变得极为顺畅 [1]。

#### 3.2 坑二：`Promise.all` 连带雪崩假死与 API 挂起
*   **踩坑现象**：在全局刷新时，我们使用了 `Promise.all(promises)`。当遇到日记本或音乐等个别应用格式解析错误或请求超时时，`Promise.all` 会瞬间整体 Reject 拒绝，导致其他本来已经生成好的数据也一并无法上屏，出现界面一直转圈假死的现象。
*   **对策**：在 `triggerGlobalRefresh` 中对并行循环应用了**单体 catch 隔离架构（Isolated Try-Catch Pipeline）**，让各个子应用生成器的 Promise 内部自行消化异常：
    ```javascript
    for (let app of listToRefresh) {
      try {
        await runSingleAppGenerator(app);
      } catch (err) {
        console.error(`应用 [${app}] 同步生成失败:`, err);
      }
    }
    ```
    即使由于掉格式或超时导致日记没有同步成功，备忘录、通讯和相册也将照常落库上屏，抗灾自愈能力极强 [1]。

#### 3.3 坑三：三维数组正则表达式带来的购物账单解析窒息
*   **踩坑现象**：在匹配购物购物车与账单的正则切除中，由于录入笔误多打了一个词：`replace(/^[\\[【]?(购物车)[\\]机制]?/gi, "")`。多出来的 `机制` 两个字导致正则完全无法正确切除标签，账单明细全部由于数组前缀未剥离而无法解析出数字，账单页面刷新后直接为空。
*   **对策**：将该 regex 纠偏并统一修正为了标准的 `[\\]】]`。修正后对 `[购物车1]` 与 `[账单5]` 的识别率在各模型表现下均极其稳定 [5]。

#### 3.4 坑四：数据管理备份导入时的 RW 事务锁死假死
*   **踩坑现象**：为了持久化，我们向 `db.js` 添加了 `check_phone_states` 新表，但在 `app_settings.js` 进行全量备份还原导入（`importBackup`）时，由于没有将新表名列入 `db.transaction('rw', [...])` 事务锁声明中，IndexedDB 事务执行到此处时由于检测到漏锁直接抛出异常并导致全手机数据导入假死、格式化中断。
*   **对策**：在 `app_settings.js` 中完美将新注册的物理数据表名加锁进 `importBackup` 读写控制中，从底层彻底根治了数据跨域死锁假死，保障了备份还原的安全持久 [2.2]。

### 2. 核心模块技术路线与实现深度解说

#### 2.1 玩家与 NPC 的单人沙箱零冲突实体判定 (`isNpc` 状态机)
*   **技术瓶颈**：在单人本地沙箱环境中，玩家账户（`forum_accounts`）的小号 ID 从 `1` 开始自增，而小号中枢（`forum_npc_accounts`）的 ID 也是从 `1` 开始。由于共享了相同的整数区间，在读取发帖人和私信发送方时，直接判等 `senderId === forumActiveAccountId` 就会将 NPC 发送的对白误判定为“我（玩家）”发出的，导致对话框和列表预览严重混乱，甚至篡改了个人主页。
*   **技术路线**：
    1.  我们在 `db.forum_messages` 消息表结构中动态追加了 **`isNpc` 布尔标识字段**。
    2.  在玩家点击发送上屏时，写入 `isNpc: false`；在 AI 分身产生应答（`forumTriggerNpcDMReply`）时，写入 `isNpc: true`。
    3.  在渲染层进行严格的安全组装：
        ```javascript
        const isSelf = !m.isNpc && Number(m.senderId) === Number(forumActiveAccountId);
        ```
        这种判定方法不依赖任何复杂的字符串 ID 拼接（如 `'npc_1'`），对数据库索引有着天然的高兼容性，彻底在物理层面上斩断了发送人主客体的识别冲突。

#### 2.2 推特极简 DM 消息舱与多媒体卡片编译器 (`forumParseMessageToCard`)
*   **Twitter DM UI 样式设计**：
    采用行内样式（`style.cssText`）进行高保真重构。对方（NPC）采用浅灰（`#eff3f4`）底色与深色（`#0f1419`）文字，左侧圆角 $18\text{px}$，右下角尖角（$4\text{px}$）；己方（User）采用推特蓝（`#1d9bf0`）底色与白色文字，右侧圆角 $18\text{px}$，左下角尖角（$4\text{px}$），在视觉上完全拉开了视差。
*   **高抗噪、双轨多媒体正则编译器**：
    由于大语言模型在输出 `[转账: 100(有偿转账)]` 等指令时，可能会随机带入空格、中英文括号（`（）`、`()`、`【】`、`［］`）等格式。系统设计了高容噪的正则流分析器：
    ```javascript
    let normalized = text.replace(/［/g, "[").replace(/］/g, "]").replace(/【/g, "[").replace(/】/g, "]");
    normalized = normalized.replace(/（/g, "(").replace(/）/g, ")");

    // 语音正则 (支持可选空格)
    const voiceRegex = /\[语音\s*[\s：:]\s*([^\]\(\)]+)\s*(?:\(\s*(\d+)\s*\))?\s*\]/i;
    // 图片正则 (支持白描画白)
    const imgRegex = /\[图片\s*[\s：:]\s*([^\]]+)\s*\]/i;
    // 转账正则 (支持可选括注留言，兼容多重空格)
    const txRegex = /\[转账\s*[\s：:]\s*([\d\.]+)\s*(?:\(\s*([^\]\)]+)\s*\))?\s*\]/i;
    ```
    通过执行该正则流，系统能顺畅地将 AI 吐出的非结构化文本，在前端毫秒级编译成精美的“微信语音卡片”、“时空模拟图卡”以及“微信橙黄色转账卡片”。

#### 2.3 双向有偿转账与物理钱包入账闭环 (`forumReceiveNPCInChat`)
*   **玩家发起转账**：
    轻触左侧加号弹出菜单，点击“有偿转账”并录入金额，自动往数据库追加 `[转账: 金额 (有偿转账)]` 的特殊标识。
*   **NPC 收款与惊喜回赠**：
    点击微信卡片时，触发 `forumReceiveNPCInChat()` 回调将金额并入系统零钱，弹出 Toast。若 NPC 处于可交互状态，在接收到用户的转账消息后，会有 $50\%$ 的概率在大模型决策下向玩家发起“回款/回礼”转账：`[转账: 惊喜回赠金额 (不客气，这是多余的报酬)]`，实现了极其逼真的双向金钱流转交互。

#### 2.4 气泡双击 Reroll（对话级联回溯重回机制）
*   **手势绑定**：双击私信气泡（`row.ondblclick`），动态截获该消息的 `createdAt` 时间戳 [2]。
*   **级联回溯删除算法**：
    在 Reroll 被激活后，系统在内存中检索当前会话中**创建时间晚于或等于该消息的所有消息集合**，并在 Dexie 事务中对其进行原子化物理擦除。
    ```javascript
    const followingMsgs = await db.forum_messages
      .where('conversationId').equals(forumSelectedConvId)
      .filter(m => m.createdAt >= target.createdAt)
      .toArray();
    for (let m of followingMsgs) {
      await db.forum_messages.delete(m.id);
    }
    ```
    擦除完成后，系统自动对剩余的最新上下文对白进行重绘，并反向调用 `forumTriggerNpcDMReply()`，使 AI 站在历史的指定节点上重新演练并推理出全新对话分支。

### 3. 踩过的坑及底层自愈性解决方案 (PAIN POINTS & SOLUTIONS)

#### 3.1 坑一：个人主页重叠声明 `isSelf` 导致的语法解析器完全锁死
*   **踩坑现象**：上一版合并中，在重绘个人空间函数 `forumInitProfileViewPage` 中，既在顶部声明了 `const isSelf = Number(userId) === ...`，又在函数中段重复用 `const isSelf` 进行了二次初始化。由于使用了 `const` 强约束，V8 引擎直接抛出 `Uncaught SyntaxError: Identifier 'isSelf' has already been declared`，导致浏览器当场拒绝加载整份 `app_forum_loader.js`。由于该主文件挂起，侧边栏按钮和账户切换在主桌面完全瘫痪、点按无响应。
*   **自愈方案**：彻底清查主装载器，删除中段冗余的二次 `const isSelf` 声明，确保函数变量生命周期单一、干净。修改后，全站的侧边栏、头像加载、Drawer 抽屉拉取瞬间满血复活，编译异常清零。

#### 3.2 坑二：Dexie `SchemaError` 引起的批量网络更新推演雪崩
*   **踩坑现象**：为实现不同账户分身的完全隔离，我们在代码中使用了 `db.forum_npc_accounts.where('userId').equals(...)`。由于玩家现有的 `db.js` Version 16 中并没有在 `forum_npc_accounts` 表的键路径（KeyPath）中注册 `userId` 的多维索引，Dexie 直接罢工报出 `SchemaError: KeyPath userId on object store is not indexed`，导致论坛冷启动与发帖推演连带锁死，卡在“正在推演时空动态...” [4]。
*   **自愈方案（绕过物理索引的内存过滤器）**：
    不改动用户的任何 `db.js` 物理表结构（防止因表版本升级产生的数据迁移损坏风险）。我们果断将索引查询重构为**“先拉全量，再使用 JavaScript 的 `filter()` 机制在内存中快速求差”**：
    ```javascript
    const myNpcs = (await db.forum_npc_accounts.toArray())
      .filter(n => Number(n.userId) === Number(forumActiveAccountId));
    ```
    对于小号总数极少的轻量扮演环境，该方法耗时仅需 $1\text{ms}$，在绕过 Dexie 物理索引报错的同时，完美实现了小号中枢在多账户间的绝对物理隔离 [4]。

#### 3.3 坑三：异步循环读取（Awaiting Loop）引发的页面高频闪屏
*   **踩坑现象**：无论是发帖 Timeline 刷新还是私信会话、关注列表，原代码普遍采用以下逻辑：
    1. 擦空容器：`container.innerHTML = ""`。
    2. 进入循环：`for (let p of posts)`。
    3. 在循环中发起异步 DB 查询：`const npc = await db.forum_npc_accounts.get(...)`。
    由于循环内存在多次 `await` 数据库锁阻滞，导致浏览器在擦除后会有长达数十毫秒的“空档期”，用户每点一次刷新、发送一条对白或者点赞，整个列表就会痛苦地白色闪动一下。
*   **自愈方案（DocumentFragment 同步交换机制）**：
    将数据库拉取与 DOM 树操纵进行绝对的**异步与同步隔离**：
    1. 在清空容器前，先将列表需要的所有关联数据（`posts`、`npcs`、`follows`）利用 `Promise.all` 或全量 `toArray()` 一次性预载入内存。
    2. 建立虚拟的内存片段：`const fragment = document.createDocumentFragment()` [2]。
    3. 在 `for` 循环中，**仅在内存中进行同步的数据匹配与节点构建（完全不使用任何 await）** [1]，并不断挂载到 fragment 上 [2]。
    4. 拼装完毕后，执行 back-to-back 的单次清空与追加：
       ```javascript
       container.innerHTML = "";
       container.appendChild(fragment);
       ```
       这一改动在物理层面上斩断了渲染空档，使帖子详情页返回和单聊发送时的闪屏现象彻底绝迹。

#### 3.4 坑四：Twitter DM 底部视口重合与气泡覆盖
*   **踩坑现象**：推特 DM 的输入底栏是 `position: absolute; bottom: 0` 绝对定位，而消息流 `#forum-chat-messages-flow` 的高度为 `calc(100% - 110px)`。在消息较多时，最底部的那一条最新气泡经常会被 58px 高度的输入栏遮挡住，且滚动不到最上方，用户只能“盲打”对白，体验很差。
*   **自愈方案**：在 `#forum-chat-messages-flow` 的行内样式中，追加强安全区内补丁 `padding: 16px 16px 80px 16px;`。通过人为在列表最下方堆叠一个 80px 的透明软边距，确保即使列表拉到最底，最新的一条气泡也永远处于绝对定位输入栏上方，解决了消息覆盖的视觉 Bug。

### 2. 核心重构组件深度剖析

#### 2.1 双轨主客体隔离与无损单人沙箱 Timeline
*   **ID 碰撞痛点**：由于单人沙箱中只有当前登录的用户是唯一的 User 账号（ID 默认从 1 开始），而引入的小号 NPC 以及自动推演出的外部网民也在 `db.forum_npc_accounts` 表中从 ID `1` 开始自增。这导致在帖子详情和评论树加载时，系统误将用户 `id: 1` 的发贴渲染成了 NPC `id: 1` 的身份，篡改了玩家的发帖与发评。
*   **反向 Timeline 隔离技术路线**：
    为了在不使用繁杂的前缀（如 `'npc_'`）前提下保障多账户的隔离与 100% 数据库兼容性：
    1.  **消息主客体识别**：我们直接通过 `m.isNpc` 属性结合 `Number(m.senderId) === Number(forumActiveAccountId)` 判断消息归属，彻底解决了私信列表方向颠倒的 Bug。
    2.  **点赞高精度对齐**：重构了 `forumToggleLike` 点赞控制器。用户每次点赞和取消都会真实向 `db.forum_likes` 表进行落库与擦除 [1]：
        ```javascript
        const myLikes = (await db.forum_likes.toArray()).filter(l => Number(l.userId) === Number(forumActiveAccountId) && l.targetType === 'post');
        const isLikedByMe = myLikes.some(l => l.targetId === p.id);
        ```
        主 Feed 在刷新载入时会高精度匹配此状态，标红点赞爱心。个人详情的“喜欢”页签亦完美对接该物理表。
    3.  **Timeline 反向隔离算法**：
        之前只提取自己和自己小号的发帖，导致大模型刷出的公共网民帖子被全部抹除。现重构为“反向隔离法”：
        ```javascript
        const otherNpcs = (await db.forum_npc_accounts.toArray())
          .filter(n => n.userId && Number(n.userId) !== Number(forumActiveAccountId))
          .map(n => n.id);
        posts = posts.filter(p => !otherNpcs.includes(Number(p.authorId)));
        ```
        只有被其他账户明确引入的小号帖子才会被隔离；而公共广场路人帖子及当前账户的 NPC 帖子则完全共享，使 Timeline 在多账户下能绝对隔离且不失活跃度 [4]。

#### 2.2 推特极简 DM 消息舱与多形态高兼容多媒体卡片编译器
*   **Twitter DM 样式重构**：
    对方（NPC）采用浅灰（`#eff3f4`）底色与深色（`#0f1419`）文字，左侧圆角 $18\text{px}$，右下角尖角（$4\text{px}$）；己方（User）采用推特蓝（`#1d9bf0`）底色与白色文字，右侧圆角 $18\text{px}$，左下角尖角（$4\text{px}$）。
*   **超强容噪、双轨多媒体正则编译器**：
    模型在输出 `[转账: 100(有偿转账)]` 等多媒体指令时，极易带入额外的空格、中英文括号。系统通过清洗和高宽容正则，对多媒体格式进行高兼容支持 [1]：
    ```javascript
    // 兼容性清洗
    let normalized = text.replace(/［/g, "[").replace(/］/g, "]").replace(/【/g, "[").replace(/】/g, "]");
    normalized = normalized.replace(/（/g, "(").replace(/）/g, ")");

    // 编译语音卡片 (支持可选空格)
    const voiceRegex = /\[语音\s*[\s：:]\s*([^\]\(\)]+)\s*(?:\(\s*(\d+)\s*\))?\s*\]/i;
    // 编译图片卡片 (支持画面白描)
    const imgRegex = /\[图片\s*[\s：:]\s*([^\]]+)\s*\]/i;
    // 编译转账卡片 (支持可选括注留言，兼容多重空格)
    const txRegex = /\[转账\s*[\s：:]\s*([\d\.]+)\s*(?:\(\s*([^\]\)]+)\s*\))?\s*\]/i;
    ```
    通过执行该正则流，系统能顺畅地将 AI 吐出的非结构化文本，在前端毫秒级编译成精美的“微信语音卡片”、“时空模拟图卡”以及“微信橙黄色转账卡片”，完美契合了微信多媒体演练。

#### 2.3 消息双击 Reroll（对话级联回溯重回机制）与加号多功能控制
*   **双击气泡唤醒管理工具**：
    双击任意消息气泡（`row.ondblclick`），即可唤醒包含“编辑 / 删除 / 重新生成 (Reroll) / 进入多选”的专业管理界面。
*   **时空线级联回溯算法 (Reroll)**：
    当用户点击重回（Reroll）时，系统不仅重滚当前消息，还会**自动向上检索并读取当前气泡之上、最近的一条 User 消息**；保留该 User 发言，并在 IndexedDB 中**物理级联删除该时间戳之后的所有 NPC 和 User 消息对白**，重新组装上下文向 API 发起请求，从而实现了真正的时空线回溯 [2]：
    ```javascript
    const prevUserMsg = await db.forum_messages
      .where('conversationId').equals(forumSelectedConvId)
      .filter(m => m.createdAt <= target.createdAt && !m.isNpc)
      .reverse().first();
    const rollbackTime = prevUserMsg ? prevUserMsg.createdAt : target.createdAt;
    
    // 级联粉碎删除
    const toDelete = await db.forum_messages
      .where('conversationId').equals(forumSelectedConvId)
      .filter(m => m.createdAt > rollbackTime).toArray();
    for (let m of toDelete) await db.forum_messages.delete(m.id);
    ```
*   **加号多媒体模拟与双向转账**：
    输入框最左侧配备了“+”功能面板。轻触可发布模拟语音、图片画白、转账消息。玩家在对 AI 转账后，对方不仅会触发收款逻辑，同时由于自动付款 timeout 模拟被彻底剥离，AI 将完全在人设和上下文驱使下产生最自然的回谢或者惊喜“回赠”转账，人情味拉满。

#### 2.4 原生 Android 特权流式分片二进制直写协议 (Android Chunked Streaming Write)
*   **特权开发背景**：当玩家的本地书架正文、备份、桌宠大图 Base64 堆叠在一起时，导出的 JSON 文本会膨胀至 40MB 左右。在此数据量下，直接通过 JavaBridge（`JavascriptInterface`）进行单次数据传输，会瞬间塞爆系统的 Binder 传输通道（安全上限仅有 1MB 左右），引发 `TransactionTooLargeException` 运行时大面积崩溃。
*   **原生分片直写（Chunked Streaming Write）技术路线**：
    我们在 Kotlin 原生端开辟了一个文件追加流句柄 `FileOutputStream`，并在 PWA 端设计了**高带宽流式切片发送协议**：
    ```
    [Web 端 PWA 线程]                                    [Android Kotlin 原生端]
    JSZip 极致压缩 JSON (40MB -> 3MB)
           │
           ▼
    编译为 Base64 编码字符串 (体积压缩 90%)
           │
           ▼
    startBinaryChunkedSave(fileName) ───────────> 清空历史、创建目标 ZIP
           │                                            并锁定物理追加流 [FileOutputStream]
           ▼
    while (offset < len) { 
       slice 500KB 字节段 (安全分片)
       appendBinaryChunk(base64Chunk) ──────────> 解码 Base64 还原为原始 ByteArray
    }                                                   直接追加直写入磁盘，保持500KB低内存
           │
           ▼
    closeBinaryChunkedSave() ───────────────────> 刷盘固化 (flush) 并安全释放文件句柄
                                                        在 Download 目录下生成标准 .zip
    ```
    该协议将混合 App 的跨端传输开销从 40MB 压实到 500KB 级别，彻底终结了任何大型数据导出时的 JVM 崩溃和 IPC 超限异常 [2]。
*   **万能自愈导入引擎**：
    在导入还原备份文件时，系统不仅完美支持网页端的 `.zip` 文件和历史的老版 `.json` 明细包，同时新增了 **Base64 自动解密自愈通道**：
    在读取 `.json` 文件时，若检测到其内容以 `UEsDB` 开头（即 ZIP 二进制魔数 `PK\x03\x04` 在 Base64 编码后的标准头部形式），系统会自动判定这是一个高压缩率的加密 JSON 包，自动在内存中进行 Base64 逆向解码、ZIP 解压并导入，保障了 100% 的向下抗灾自愈能力 [2]。

#### 2.5 列表无锁单次 DOM 渲染与防闪屏策略
*   **闪动成因**：先前的发帖 Feed 时间线、私信会话、关注列表和聊天记录在刷新时，都是先简单粗暴地执行 `innerHTML = ""` 清空，再去循环 Await 轮询数据库头像。这导致列表在渲染时会出现数十毫秒的“空档期”，使用户体感到严重的闪动。
*   **自愈方案（DocumentFragment 同步交换机制）**：
    我们对这四处列表渲染进行了彻底的重构：在清空容器前，先将列表需要的所有关联数据（`posts`、`npcs`、`follows`）一次性预载入内存。利用 `document.createDocumentFragment()`，在内存中进行同步的数据匹配与节点构建（完全不使用任何 await） [1, 2]。拼装完毕后，执行单次清空与追加。这一改动在物理层面上斩断了渲染空档，使闪屏现象彻底绝迹 [1]。

### 3. 踩过的坑及底层自愈性解决方案 (PAIN POINTS & SOLUTIONS)

#### 3.1 坑一：个人空间主页重叠声明 `isSelf` 导致的语法解析器完全锁死
*   **踩坑现象**：上一版合并中，在重绘个人空间函数 `forumInitProfileViewPage` 中，既在顶部声明了 `const isSelf = Number(userId) === ...`，又在函数中段重复用 `const isSelf` 进行了二次初始化。这直接导致浏览器拒绝编译并挂起了整个 `app_forum_loader.js` 的加载。由于主入口失效，侧边栏、账户切换在主桌面完全瘫痪。
*   **自愈对策**：彻底清查主装载器，删除中段冗余的二次 `const isSelf` 声明，确保函数变量生命周期单一。修改后，全站的侧边栏、头像加载、Drawer 抽屉拉取瞬间复活。

#### 3.2 坑二：私信多媒体和文字混合发送时的截断异常
*   **踩坑现象**：先前由于多媒体卡片（如语音、转账）的解析是以单条消息为单位的，如果大模型返回的文本是 `最近降温了 [语音:要注意身体(5)] 记得带伞`，系统的正则解析会直接被卡片指令截断，导致后半段的“记得带伞”等文字直接被抛弃并丢失，甚至卡片也渲染不出。
*   **自愈对策**：开发了专用的多模私信对白分发切片器 `splitAiResponse`：
    ```javascript
    // 优先切分并拆出多媒体指令，使 [语音] 等消息完全单独作为一整条私信入库
    const regex = /(\[[^\]]+\])/g;
    const parts = text.split(regex).map(p => p.trim()).filter(Boolean);
    ```
    它能把混合文本精确剪切为多个独立的物理消息记录，顺序追加进 IndexedDB。这不仅保证了多媒体卡片的 100% 完整无缺，更让 AI 回复天然形成微信式分多条消息连续弹出的高度拟真感 [2]！

#### 3.3 坑三：私信 prompt 过于关注“活人网络黑话”导致角色崩坏 (OOC)
*   **踩坑现象**：原本的私信提示词过度诱导 AI 输出例如“😅”、“笑死”、“绝了”等现代互联网黑话，这直接导致一些档案馆中本该是高冷、仙侠、古风的引入角色也开始频繁在私信中使用网络烂梗，产生了严重的“出戏”和人设崩坍。
*   **自愈对策**：重构了 `forumTriggerNpcDMReply` 的引导机制。将“网络黑话”降为次要地位，把**“必须深度阅读并紧密跟手呼应上下文”和“绝对100%保全原设性格防线，宁可少说话也绝对不准OOC”作为最高限制底线**写入 Prompt。优化后，角色说话完全找回了原设神韵，傲娇、隐忍或冷感细节被完美保留。

### 4. 给后续开发的参考经验 (FUTURE SCALABILITY)

1.  **“主权防火墙”对 AI 幻觉的防御价值**：
    在本次重构中，我们在发帖、评论、私信生成器最底层阻断了 `postData.nickname === userNick` 冒名发帖。后续如果您在开发朋友圈、深谈或者相册等其他交互时，若遇到 AI 自作主张盗用玩家昵称发言的幻觉，应统一参照此方案，建立主权名字防火墙，在数据落库（add）之前进行单体 `continue` 过滤，这是最轻量且牢固的安全盾牌。
2.  **“多端同步”的 Database Transaction 事务锁安全**：

### 一、 核心功能板块与技术路线设计

#### 1. Android 系统级多媒体控制卡片（MediaSession & MediaStyle）
*   **开发背景**：原本在 Android 原生端（Kotlin）的音乐播放功能只是静默后台播放，在手机锁屏界面、状态栏和通知下拉栏中没有任何提示，无法像主流音乐软件一样直接调节进度或进行快捷控制。
*   **技术路线**：
    *   利用 Android 原生的 `android.media.session.MediaSession` 和 `android.media.session.PlaybackState` 框架接管音乐播放器的全局上下文。
    *   通过动态注册广播接收器（`BroadcastReceiver`）拦截通知栏操作发出的 `PendingIntent`。
    *   将 `McpForegroundService` 原本普通的前台通知，动态就地升级为符合 Android 10+ 规范的 **`MediaStyle` 多媒体通知控制卡片**。
    *   通过向系统报告 `mediaPlayer` 的播放进度（Position）和总长（Duration），让系统 UI 自动渲染可自由拖动的**多媒体进度条（Seekbar）**，实现与系统控件的双向控制绑定。
    *   当音乐结束或手动停止时，通知卡片顺畅地变回默认的保活守护通知，防止内存泄漏。

#### 2. Supabase PWA 多设备登录限制与安全激活码系统（Limit-2 & FIFO Queue）
*   **开发背景**：为了保障纯前端 PWA 在公开推广时的分发控制，需要建立账号密码与设备绑定制度，限制同一账号最多只能同时登录 2 台设备。数据依然安全地保存在本地 Dexie 数据库中。
*   **技术路线**：
    *   **激活码（邀请码）触发器验证**：在 Supabase PostgreSQL 数据库底层挂载 `before insert on auth.users` 触发器 `handle_new_user_with_invite_code()`。若非 `service_role`（管理员后台），新用户注册必须在 Metadata 中夹带合法的、未被使用的激活码（存储在 `public.activation_codes` 表中），否则直接中止事务，从底层根除前端被篡改绕过的风险。
    *   **设备排队队列（FIFO）**：在数据库中创建 `public.user_devices` 关系表。每个浏览器在登录时，均会在本地生成并存储唯一的 `myClientToken`。
    *   用户登录成功后，前端将调用 `verifyDeviceSession`，按时间从新到老查询用户所有设备。若总数超过 2 台，则利用 PostgreSQL 的原子删除命令直接删除最老的设备会话（先进先出）。
    *   **毫秒级强制下线（Kick-out）**：利用 Supabase Realtime 通道，订阅针对该用户 `user_id` 的 `DELETE` 变更。一旦当前设备的 Token 遭到删除，便立即在前端触发 `handleKickOut()`，清空凭证，弹窗警告并退回登录层。
    *   **15秒心跳兜底**：为了应对 WebSocket 断线等网络波动异常，设立 15 秒心跳轮询验证，确保双端强行下线的可靠性。
    *   **离线放过支持**：通过 `navigator.onLine` 嗅探网络。若无网络，则直接放过，允许用户本地正常看书与专注，不影响离线使用体验。

#### 3. 设置中心账号管理控制面板
*   **开发背景**：用户在登录后需要有直观的界面去查看当前在线的设备、查看自己的账户信息，并支持主动退登。
*   **技术路线**：
    *   在 `index.html` 的设置一级菜单中开辟“账号安全管理”选项。
    *   在 `app_auth.js` 中，当用户登录成功时，将用户密码临时加密缓存在本地（由于 Supabase 安全机制默认不返回明文密码，此举能方便用户查看记下的密码），并在下线时自动抹除。
    *   在 `app_settings.js` 中，当切入二级账号面板时，异步拉取 `user_devices` 中绑定该 `user_id` 的行数，将当前在外的总在线设备数即时显示在 UI 界面上。

#### 4. UI 层级冲突自愈（Toast / Alert 覆盖冲突）
*   由于全局登录遮罩层 `#auth-login-overlay` 的 `z-index` 设为了最高级 `99999`，此前在登录界面时，用户无法看到因登录失败、注册成功或被踢下线而弹出的 `showCustomAlert` 和 `showToast`（此前 z-index 仅为 10000，导致显示在登录层下方）。
*   通过将 `.pwa-toast` 与 `.pwa-modal-overlay` 的 `z-index` 属性分别调整为 `100001` 和 `100000`，使所有的原生弹层可以完美压在全屏登录遮罩的上方，解决交互盲区。
